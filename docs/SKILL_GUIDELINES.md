# 技能开发指南

> 如何为 claude-code-java 设计、验证和实现新技能

## 目的

本指南帮助贡献者创建高质量的技能：
- 填补真正的空白（不重复现有功能）
- 提供清晰、可操作的价值
- 与现有技能生态系统良好集成

---

## 开始之前：验证清单

在实现新技能之前，回答这些问题：

### 1. 它是否已存在？

检查与现有技能的重叠：

| 现有技能 | 级别 | 重点 |
|----------------|-------|-------|
| clean-code | 微观 | 函数、命名、DRY/KISS |
| solid-principles | 微观/中观 | 类、接口 |
| design-patterns | 中观 | 类协作模式 |
| java-code-review | 中观 | 代码审查清单 |
| architecture-review | 宏观 | 包、模块、层 |
| spring-boot-patterns | 框架 | Spring Boot 特定内容 |
| jpa-patterns | 框架 | JPA/Hibernate 特定内容 |
| security-audit | 横切 | OWASP、输入验证 |
| test-quality | 横切 | JUnit 5、AssertJ |

**问自己：**
- 现有技能是否覆盖了我想要添加的 >50% 的内容？
- 扩展现有技能是否比创建新技能更好？

### 2. 它在什么级别运行？

技能应该有清晰的范围：

```
┌─────────────────────────────────────────────────┐
│  宏观    │ 包、模块、架构     │
├───────────┼─────────────────────────────────────┤
│  中观     │ 类、接口、协作  │
├───────────┼─────────────────────────────────────┤
│  微观    │ 函数、变量、表达式   │
├───────────┼─────────────────────────────────────┤
│  框架│ Spring、JPA、特定技术  │
├───────────┼─────────────────────────────────────┤
│  横切    │ 安全、测试、日志          │
└───────────┴─────────────────────────────────────┘
```

**警告信号：** 试图覆盖多个级别的技能可能太宽泛了。

### 3. 它是"审计"还是"模板"？

两种类型的技能：

| 类型 | 目的 | 示例 |
|------|---------|---------|
| **审计** | 审查现有代码，发现问题 | java-code-review、security-audit |
| **模板** | 展示如何编写新代码 | spring-boot-patterns、design-patterns |

**问自己：**
- 我的技能是哪种类型？
- 同类型的现有技能是否已经覆盖了这一点？

示例：
- `spring-boot-patterns` = 模板（如何编写）
- `api-contract-review` = 审计（检查现有 API）
- 两者都处理 REST API，但服务不同的目的 ✅

### 4. 它添加了什么独特价值？

技能应该添加新的东西：

**新技能的好理由：**
- 覆盖现有技能未涉及的主题
- 相同主题但不同级别（微观 vs 宏观）
- 相同主题但不同类型（审计 vs 模板）
- 不适合通用技能的专业深度

**坏理由：**
- "有它会很好"
- "其他工具有这个"（没有验证空白）
- "我想重新组织现有内容"

### 5. 它是否足够专注？

一个技能应该在一个会话中可完成。太宽泛的迹象：
- 超过 10-15 个清单项
- 覆盖多个不相关的主题
- 完全应用需要数小时

**解决方案：** 拆分为多个专注的技能。

## 技能解剖结构

每个技能有两个文件：

```
.claude/skills/<skill-name>/
├── SKILL.md    # Claude 的指令（AI 读取此文件）
└── README.md   # 人类的文档
```

### SKILL.md 结构

```markdown
---
name: skill-name
description: 一行描述。何时使用 [触发器]。
---

# Skill Name

简要介绍（1-2 句话）。

## 何时使用
- 触发短语 1
- 触发短语 2

## 快速参考
[表格或摘要用于快速查找]

## 主要内容
[清单、模式、示例]

## Token 优化
[如何在大型代码库上高效使用]
```

### README.md 结构

```markdown
# Skill Name

> 一行标语

## 它做什么
[2-3 句话]

## 何时使用
[带有示例短语的项目符号]

## 核心概念
[主要思想的简要解释]

## 使用示例
[显示典型交互]

## 相关技能
[互补技能的链接]

## 参考资料
[外部资源]
```

## 质量标准

### 内容指南

1. **针对 Java**
   - 在示例中使用 Java 语法
   - 引用 Java 生态系统工具（Maven、JUnit、Spring）
   - 不要通用 — 这是其他技能集合所做的

2. **提供可操作的清单**
   - ✅ `- [ ] 使用 Optional 检查空值安全`
   - ❌ `- [ ] 确保代码良好`

3. **展示反模式**
   ```java
   // ❌ 坏：为什么这是错的
   badCode();

   // ✅ 好：为什么这是对的
   goodCode();
   ```

4. **包含严重性级别**
   - 高：Bug、安全问题、性能杀手
   - 中：代码异味、可维护性问题
   - 低：风格、次要优化

### Token 效率

技能应该帮助 Claude 高效工作：

1. **优先检查** - 最重要的在前
2. **提供命令** - 用于快速收集信息的 Shell 命令
3. **建议采样** - "检查 2-3 个示例，而不是每个文件"
4. **提前退出** - "如果 X，跳过 Y 检查"

## 验证过程

在提交新技能之前：

### 步骤 1：空白分析

```markdown
## 提议：<skill-name>

### 它做什么
[描述]

### 重叠检查
| 现有技能 | 重叠？ | 注意 |
|----------------|----------|-------|
| skill-1 | 无/部分/高 | ... |
| skill-2 | 无/部分/高 | ... |

### 独特价值
[这添加了什么不存在的]

### 结论
[ ] 无显著重叠 - 继续
[ ] 部分重叠 - 记录区别
[ ] 高重叠 - 考虑扩展现有技能
```

### 步骤 2：同行评审问题

问自己（或审查者）：

1. 我会在真实项目中实际使用这个技能吗？
2. 我能否用一句话解释何时使用它 vs 相关技能？
3. 清单是否少于 15 项？
4. 所有示例都是 Java 的（不是伪代码）吗？

### 步骤 3：真实代码测试

在提交之前：
- 将技能应用于真实的 Java 项目
- 验证清单有意义
- 检查建议是否可操作

### 自动化审查

修改 `.claude/skills/` 的 PR 会根据这些指南自动审查。
审查检查：
- **结构**：frontmatter、必需文件、文件夹约定
- **重叠**：与现有技能的比较
- **质量**：可操作的内容、Java 特定示例、专注的范围

此自动检查在人工审查之前运行，以便及早发现常见问题。

## 要避免的反模式

### 1. 全能型

❌ **坏：** 试图覆盖所有内容的技能
```
java-best-practices/
  - 覆盖代码风格
  - AND 测试
  - AND 架构
  - AND 安全
  - AND 性能
```

✅ **好：** 专注的技能，组合良好

### 2. 重叠陷阱

❌ **坏：** 当 `java-code-review` 已经有异常部分时创建 `exception-handling-review`

✅ **好：** 扩展 `java-code-review` 或确保新技能有明显不同的范围

### 3. 复制粘贴技能

❌ **坏：** 从互联网复制通用建议

✅ **好：** Java 特定的、有观点的、实用的指导

### 4. 理论技能

❌ **坏：** 在没有可操作检查的情况下解释概念

✅ **好：** 可以立即应用的清单

## 扩展 vs 创建

有时扩展现有技能更好：

### 何时扩展

- 添加 <5 个新清单项
- 与现有技能相同的级别和类型
- 自然适合现有结构

### 何时创建新的

- 不同级别（微观 vs 宏观）
- 不同类型（审计 vs 模板）
- 会使现有技能太长（>200 行）
- 不同的触发短语

### 如何扩展

1. 彻底阅读现有技能
2. 确定新内容适合的位置
3. 添加新部分或扩展现有部分
4. 如需要更新 README
5. 提交并附带清晰消息：`enhance(skill-name): add X checks`

---

## 提交约定

对于技能更改：

```
feat: add <skill-name> skill          # 新技能
enhance(<skill-name>): add X checks   # 扩展现有技能
fix(<skill-name>): correct Y example  # 修复问题
docs(<skill-name>): improve Z section # 仅文档
```

## 迭代改进

技能通过实际使用不断改进。第一个版本后技能永远不会"完成"。

### 技能需要精炼的信号

| 信号 | 问题 | 解决方案 |
|--------|---------|----------|
| Claude 问澄清问题 | 缺少上下文或默认值 | 添加明确的默认值和示例 |
| 你经常更正输出 | 缺少约束 | 添加"不要"规则或格式规范 |
| 输出变化太大 | 太模糊 | 添加预期输出的具体示例 |
| 在一个项目上工作，在另一个项目上失败 | 太窄 | 概括模式，添加边缘情况 |
| 需要多次迭代才能正确 | 缺少结构 | 添加分步工作流 |

### 改进工作流

1. 在真实工作**上使用技能**
2. **记下你对 Claude 输出的每次更正**
3. **用更正作为规则更新 SKILL.md**
4. **清除上下文**并再次测试
5. **重复 5-6 次**直到稳定

### 成熟度指标

一个技能在以下情况下成熟：
- 首次尝试 >80% 的时间有效
- 不需要引导或更正
- 输出可复制粘贴
- 跨不同项目工作

### 可选：跟踪迭代

考虑添加到你的技能文件夹：

```
.claude/skills/<skill-name>/
├── SKILL.md
├── README.md
└── CHANGELOG.md   # 可选：跟踪精炼
```

示例 CHANGELOG.md：
```markdown
## 迭代历史
- v1：初始版本，太模糊
- v2：添加了输出格式规范
- v3：在常见错误后添加了"避免"列表
- v4：添加了边缘情况示例
- v5：稳定 - 在 3 个项目上测试
```

## 清单摘要

在提交新技能之前：

- [ ] 检查与所有现有技能的重叠
- [ ] 确定清晰的级别（微观/中观/宏观/框架/横切）
- [ ] 确定类型（审计 vs 模板）
- [ ] 记录添加的独特价值
- [ ] SKILL.md 遵循结构约定
- [ ] README.md 提供人类友好的文档
- [ ] 所有示例都是 Java 特定的
- [ ] 清单有 <15 个可操作项
- [ ] 在真实代码上测试
- [ ] 更新 skills/README.md 表
- [ ] 如需要更新主 README.md
